## 一、设计原则

1. **按文件类型分层**：配置 → `deploy/`，代码 → `src/`，文档 → `docs/`
2. **代码按功能子模块组织**：`src/` 下按 workflow / tools / knowledge / tests 分子目录
3. **敏感文件隔离**：`.env`（含真实 Key）不入库，`.env.template` 可入库
4. **数据文件分离**：PDF 手册、元数据 JSON、测试报告等生成物单独归档
5. **根目录保持简洁**：只放 README、.gitignore、规范文件

## 三、目录用途说明

| 目录 | 用途 |
|------|------|
| `deploy/` | 存放部署相关配置：Docker Compose 扩展、环境变量模板、启动/验证脚本 |
| `src/` | 存放所有可执行源代码和结构性配置文件（DSL、Schema、Prompt） |
| `src/workflow/` | Dify Chatflow 的 DSL 配置和 System Prompt 文本 |
| `src/tools/` | 自定义工具的 OpenAPI Schema 和封装代码 |
| `src/knowledge/` | 知识库构建相关脚本：PDF 预处理、检索验证 |
| `src/tests/` | 自动化测试脚本：场景验收、性能压测 |
| `docs/` | 存放所有 Markdown 文档：PRD、操作指南、调试手册、验收清单 |
| `data/` | 存放原始数据文件（PDF 手册）和预处理输出（元数据 JSON），不入库 |
| `reports/` | 存放脚本自动生成的报告文件，不入库 |

# 制造业设备维修知识库智能问答平台

> 基于 Dify + Pipeline RAG 的制造业设备维修智能问答 Demo

本项目是一个演示型 Demo，验证 Dify Workflow + RAG 技术栈在制造业设备维修垂直场景的可行性。核心能力：预置知识库优先检索 + 联网搜索兜底的混合检索策略，配合结构化输出和引用溯源。

## 快速开始

### 1. 环境准备

```bash
# 复制环境变量模板
cp deploy/.env.template .env

# 编辑 .env，填入 API Key
#   DEEPSEEK_API_KEY  → https://platform.deepseek.com 获取
#   TAVILY_API_KEY    → https://tavily.com 获取
```

### 2. 启动服务

```bash
# Linux/Mac/WSL2
chmod +x deploy/startup.sh
./deploy/startup.sh

# Windows PowerShell
.\deploy\startup.ps1
```

### 3. 验证环境

```bash
./deploy/verify-setup.sh
```

### 4. 配置 Dify

按 `docs/部署检查清单.md` 完成 Dify Web UI 的 5 步配置。

### 5. 运行测试

```bash
# 知识库检索验证
python src/knowledge/verify_retrieval.py --dataset_id xxx --api_key dataset-xxx

# 自动化验收测试
python src/tests/test_scenarios.py --api_key app-xxx
```

## 七、额外建议

### 7.1 建议新增的文件类型

| 文件 | 建议路径 | 理由 |
|------|---------|------|
| `CHANGELOG.md` | 根目录 | 记录版本变更，方便团队追踪 |
| `Makefile` | 根目录 | 封装常用命令（make start / make test / make verify） |
| `requirements.txt` | `src/` | Python 依赖清单（requests, PyPDF2 等） |
| `docker-compose.local.yaml` | `deploy/` | 本地开发用 override（区别于演示用） |

### 7.2 System Prompt 格式建议

将 `System-Prompt最终版.md` 改为 `system-prompt.txt`（纯文本）：
- **理由**：System Prompt 需要直接粘贴到 Dify LLM 节点，纯文本格式避免 Markdown 标记干扰
- **保留 .md 版本**：如需带注释的说明版，可额外保留 `docs/system-prompt-说明.md`

### 7.3 报告文件管理

| 报告类型 | 生成脚本 | 输出路径 | 入库? |
|---------|---------|---------|-------|
| 测试报告 | `src/tests/test_scenarios.py` | `reports/测试报告.md` | ❌ 不入库 |
| 检索验证报告 | `src/knowledge/verify_retrieval.py` | `reports/检索验证报告.md` | ❌ 不入库 |
| 预处理汇总报告 | `src/knowledge/preprocess_pdfs.py` | `reports/预处理汇总报告.md` | ❌ 不入库 |

**理由**：报告是自动生成的，可随时重新生成，入库会造成无意义的 diff 噪声。

### 7.4 Dify 克隆目录处理

`dify/` 目录是 startup 脚本自动克隆的 Dify 官方仓库，**不入库**：
- 已加入 `.gitignore`
- 脚本自动检测：不存在则克隆，存在则跳过
- 如需修改 Dify 配置，通过 `deploy/docker-compose.override.yaml` 扩展，不直接改 dify/ 源码
