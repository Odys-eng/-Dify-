# 故障应急卡

> **文件路径**：`docs/故障应急卡.md`
> **用途**：演示前检查清单 + 常见故障 30 秒应急处理
> **使用场景**：Demo 演示前的最后检查 + 演示中突发故障的快速恢复

---

## 一、演示前检查清单（必查 5 项）

> ⏰ **演示前 10 分钟执行**，全部打勾后方可开始演示

| # | 检查项 | 检查方法 | 通过标准 | 状态 |
|---|--------|---------|---------|------|
| 1 | Docker 容器全部运行 | `cd dify/docker && docker compose ps` | 7 个容器全部 running | ☐ |
| 2 | Dify Web UI 可访问 | 浏览器打开 http://localhost | 显示登录/控制台页面 | ☐ |
| 3 | DeepSeek API 可用 | 发送测试消息 | 5 秒内返回正常回答 | ☐ |
| 4 | Tavily API 可用 | 检查免费额度 ≥ 800 次 | https://tavily.com 登录查看 | ☐ |
| 5 | 知识库检索正常 | 在 Dify 知识库「召回测试」输入查询 | 返回相关片段，score ≥ 0.5 | ☐ |

### 快速检查命令

```bash
# 一键检查 Docker 容器状态
cd dify/docker && docker compose ps --format "table {{.Name}}\t{{.Status}}"

# 检查 Dify API 可访问性
curl -s -o /dev/null -w "%{http_code}" http://localhost/apps

# 检查 SiliconFlow API（从 .env 读 Key）
SILICONFLOW_API_KEY=$(grep "^SILICONFLOW_API_KEY=" dify/docker/.env | cut -d'=' -f2-)
curl -s https://api.siliconflow.cn/v1/chat/completions \
  -H "Authorization: Bearer $SILICONFLOW_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"model":"deepseek-ai/DeepSeek-V4-Pro","messages":[{"role":"user","content":"ping"}],"max_tokens":5}'
```

## 二、常见演示故障及应急处理

### 故障 1：API Key 失效

| 项 | 内容 |
|----|------|
| **症状** | 演示时返回 HTTP 401，或答案为空 |
| **可能原因** | DeepSeek 或 Tavily API Key 过期/额度耗尽 |
| **应急处理** | 30 秒内替换 Key |
| **操作步骤** | 1. 打开 Dify → 设置 → 模型供应商 → DeepSeek<br>2. 更新 API Key<br>3. 如 Tavily 失效，工具 → 自定义 → Tavily → 更新 Key |
| **预防** | 演示前用 `verify-setup.sh` 检查 Key 有效性 |
| **最坏情况** | Key 无法立即获取 → 改为展示已录制的演示视频 |

### 故障 2：Docker 容器挂掉

| 项 | 内容 |
|----|------|
| **症状** | Web UI 无法访问，或 API 返回连接超时 |
| **可能原因** | 内存不足导致 OOM，或容器异常退出 |
| **应急处理** | 30 秒内重启容器 |
| **操作步骤** | 1. 执行：`cd dify/docker && docker compose restart`<br>2. 等待 15-20 秒<br>3. 验证：`docker compose ps` 全部 running<br>4. 刷新浏览器 |
| **预防** | 演示前关闭其他大型程序（浏览器多余标签、IDE 等） |
| **最坏情况** | 重启失败 → `docker compose down && docker compose up -d`（需 1-2 分钟） |

### 故障 3：网络断连

| 项 | 内容 |
|----|------|
| **症状** | DeepSeek/Tavily API 超时，答案返回为空或报错 |
| **可能原因** | WiFi 断连、公司网络限制、DNS 故障 |
| **应急处理** | 30 秒内切换网络 |
| **操作步骤** | 1. 开启手机热点<br>2. 电脑连接热点<br>3. 验证：`curl -s https://api.siliconflow.cn/v1`<br>4. 重新发送查询 |
| **预防** | 演示前准备手机热点，测试可用性 |
| **最坏情况** | 网络完全不可用 → 改为演示场景 1（知识库命中，不依赖外网）<br>注意：知识库检索仍需 DeepSeek API，如完全断网需展示录屏 |

### 故障 4：知识库检索无结果

| 项 | 内容 |
|----|------|
| **症状** | 场景 1 查询返回兜底话术「现有资料无法完全解答」 |
| **可能原因** | score_threshold 过高、知识库未正确配置、切片问题 |
| **应急处理** | 临时切换演示顺序 |
| **操作步骤** | 1. 跳过场景 1，先演示场景 3（多轮追问）<br>2. 在 Dify 知识库设置中降低 score_threshold（0.5 → 0.3）<br>3. 重新测试场景 1<br>4. 如仍失败，改用知识库「召回测试」直接演示检索效果 |
| **预防** | 演示前用 `verify_retrieval.py` 测试 10 个查询的命中率 |
| **最坏情况** | 知识库完全不可用 → 改为演示场景 2（联网兜底），强调「联网搜索能力」 |

### 故障 5：LLM 输出格式错误

| 项 | 内容 |
|----|------|
| **症状** | 答案缺少「故障可能原因」「排查步骤」结构，或无引用 |
| **可能原因** | System Prompt 未生效、temperature 过高 |
| **应急处理** | 重新发送查询（LLM 有随机性） |
| **操作步骤** | 1. 重新发送相同查询（80% 概率恢复正常）<br>2. 如仍异常，在 Dify Workflow 中检查 LLM 节点的 Prompt<br>3. 如有权限，降低 temperature（0.3 → 0.1） |
| **预防** | 演示前用 `test_scenarios.py` 跑 3 次确保稳定 |
| **最坏情况** | 格式持续异常 → 展示已录制的正常输出截图 |

### 故障 6：响应超时（>15 秒）

| 项 | 内容 |
|----|------|
| **症状** | 演示时等待超过 15 秒，观众开始不耐烦 |
| **可能原因** | reasoning_effort=high 太慢、网络延迟、服务器负载高 |
| **应急处理** | 切换到简单查询 |
| **操作步骤** | 1. 取消当前查询<br>2. 改用简单查询（如「主轴异响」代替长描述）<br>3. 或演示场景 1（知识库命中，通常更快） |
| **预防** | 演示前关闭思考模式或降低 reasoning_effort |
| **最坏情况** | 持续超时 → 展示录屏，口头解释「正常使用中响应时间约 5-8 秒」 |

## 三、应急处理优先级

```
演示中突发故障
       │
       ▼
  能否 30 秒内恢复？
       │
   ┌───┴───┐
   ▼       ▼
  能       不能
   │       │
   ▼       ▼
 快速修复   切换方案
 (按上方    (改演示顺序
  故障表)   或展示录屏)
       │
       ▼
  继续演示
```

### 切换方案优先级

| 优先级 | 方案 | 说明 |
|--------|------|------|
| 1 | 修复后继续 | 30 秒内能修复的，优先修复 |
| 2 | 调整演示顺序 | 跳过失败场景，先演示成功的 |
| 3 | 简化查询 | 用更短的查询替代复杂查询 |
| 4 | 展示录屏 | 最后手段，提前准备录屏视频 |
