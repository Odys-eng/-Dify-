# 基于 RuyiDify 的制造业设备维修知识库问答平台设计

## 1. 项目目标

在一周内完成可用于课程答辩的制造业设备维修知识库智能问答平台。项目直接复用 RuyiDify 的完整 Dify 1.15.0 能力和现有品牌，不接入真实硬件，不建设生产级企业系统。

首期成功标准是：平台可稳定运行，能够围绕制造业设备维修资料进行有来源的问答，能够展示知识入库、检索、生成和评测闭环，并具备可复现的代码与答辩材料。

## 3. 关键边界

- 原 RuyiDify 目录始终只读，不在其中编辑、生成、提交或清理文件。
- 仅复制原仓库当前 `HEAD` 中受 Git 管理的文件。
- 不复制原 `.git`、`.env`、`key.txt`、Docker 数据卷、缓存、构建产物和未跟踪课程材料。
- 现有 Docker 与模型环境直接复用，不启动第二套同端口服务。
- 保留 RuyiDify 品牌，不进行 Logo、名称或主题替换。
- 不做硬件、PLC、IoT、工单系统、复杂权限、预测性维护和生产高可用。

## 4. 系统设计

平台沿用 RuyiDify 原生架构：浏览器访问 Dify Web，问题进入维修问答 Chatflow；Chatflow 调用知识检索节点完成召回和可选重排序，随后由大模型生成结构化维修建议并返回来源引用。

知识入库继续使用 Dify Dataset：资料上传后由 Worker 异步完成抽取、分段、Embedding 和向量存储。管理员通过原生控制台查看文档、分段、索引状态和命中测试。

维修回答统一包含：问题判断、可能原因、排查步骤、处理建议、安全注意事项和参考资料。知识库证据不足时必须明确说明资料不足，不生成确定性维修结论。

## 10. 风险控制

- 复制遗漏：以 Git `HEAD` 归档为唯一源码清单，并比较文件数量。
- 凭据泄露：不复制未跟踪文件和运行配置，提交前执行敏感信息扫描。
- 端口冲突：只复用已运行环境，不在新目录启动重复 Compose 栈。
- 索引失败：展示索引状态，准备 Markdown 小文件和已索引资料作为备用。
- 模型服务不稳定：提前录制完整演示视频，并保留标准问答截图。
- 范围膨胀：第 5 天冻结功能，后续只修复影响答辩的问题。
