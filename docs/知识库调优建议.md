# 知识库调优建议

> **文件路径**：`docs/知识库调优建议.md`
> **用途**：基于 PRD 中 score_threshold=0.5 的设定，提供知识库检索调优策略
> **适用场景**：运行 `verify_retrieval.py` 后，根据命中率调整参数

---

## 调优决策框架

```
运行 verify_retrieval.py
        │
        ▼
   命中率是多少？
        │
   ┌────┴────┐
   ▼         ▼
< 60%      ≥ 60%
   │         │
   ▼         ▼
命中率低   命中率可接受
（召回不足）  │
   │      检查误命中率
   │         │
   ▼    ┌────┴────┐
调整召回   ▼         ▼
参数     高        低
        │         │
        ▼         ▼
      调精度    状态良好
      参数
```

## 一、命中率太低（< 60%）的调优策略

### 1.1 诊断方向

命中率低通常意味着**召回不足**——知识库里有相关内容，但检索不到。按以下顺序排查：

| 排查项 | 检查方法 | 典型问题 |
|--------|---------|---------|
| 切片质量 | 查看未命中查询的 Top 1 片段内容 | 切片过大/过小，语义被稀释 |
| Embedding 适配 | 检查查询与命中文档的语言一致性 | 中英文混排导致向量空间不匹配 |
| Score 阈值 | 查看未命中查询的实际 Score | 阈值 0.5 过高，截断了相关结果 |
| 分段方式 | 检查 PDF 是否有书签结构 | 无书签导致按固定长度切，破坏语义 |

### 1.2 调整参数（按优先级排序）

#### 优先级 1：降低 Score 阈值

| 参数 | 当前值 | 建议调整 | 理由 |
|------|--------|---------|------|
| `score_threshold` | 0.5 | **0.3 - 0.4** | bge-large-zh 的分数分布偏低，0.5 可能过严 |

**操作方式**：
- Dify 知识库设置 → 检索设置 → Score 阈值
- 或修改 `verify_retrieval.py` 中的 `score_threshold` 参数重新测试

**风险**：降低阈值会引入更多噪声（误命中率上升），需配合 Rerank 使用。

#### 优先级 2：调整切片策略

| 参数 | 当前值 | 建议调整 | 适用场景 |
|------|--------|---------|---------|
| `chunk_size` | 500 | **300 - 400** | 查询多为短句（如「SP-2003 报警」），小切片更精准 |
| `overlap` | 50 | **80 - 100** | 章节边界信息丢失严重时增大 |
| 分段方式 | 按章节标题 | **通用 + 自定义分隔符** | PDF 无书签时 |

**操作方式**：
- Dify 知识库 → 文档设置 → 重新分段
- ⚠️ 重新分段会重新向量化，耗时较长

#### 优先级 3：切换检索模式

| 当前模式 | 建议切换 | 理由 |
|---------|---------|------|
| 混合检索 | **全文检索为主** | 故障代码（SP-2003）是精确关键词，全文检索更精准 |

**权衡**：全文检索对语义查询（如「主轴异响原因」）效果差，需根据查询类型决定。

#### 优先级 4：扩充知识库内容

| 场景 | 操作 |
|------|------|
| 某设备类型完全未命中 | 补充该设备的手册/案例文档 |
| 某故障类型覆盖不足 | 补充故障案例集或维修记录 |
| 中英文混排召回差 | 补充中英文对照术语表 |

## 二、误命中率太高（score ≥ 0.5 但内容不相关）的调优策略

### 2.1 诊断方向

误命中率高意味着**精度不足**——检索到的片段分数高，但内容与查询无关。通常原因：

| 原因 | 表现 | 解决方向 |
|------|------|---------|
| Score 阈值过低 | 大量边缘结果被返回 | 提高 `score_threshold` |
| Rerank 未启用 | 原始向量相似度高但语义不相关 | 开启 Rerank |
| 切片过大 | 单个切片包含多个主题 | 减小 `chunk_size` |
| Embedding 模型不匹配 | 通用模型对专业术语理解差 | 换用专业领域 Embedding |

### 2.2 调整参数（按优先级排序）

#### 优先级 1：开启并优化 Rerank

| 参数 | 当前值 | 建议调整 | 理由 |
|------|--------|---------|------|
| `reranking_enable` | True | **确认已开启** | Rerank 对精度提升显著 |
| Rerank 模型 | 默认 | **bge-reranker-base** | 中文场景推荐 |

**操作方式**：
- Dify 知识库 → 检索设置 → Rerank → 开启 → 选择 `bge-reranker-base`

#### 优先级 2：提高 Score 阈值

| 参数 | 当前值 | 建议调整 | 理由 |
|------|--------|---------|------|
| `score_threshold` | 0.5 | **0.6 - 0.7** | 过滤掉边缘命中，只保留高相关结果 |

**风险**：提高阈值会降低召回率，需在精度和召回之间权衡。

#### 优先级 3：减小切片大小

| 参数 | 当前值 | 建议调整 | 理由 |
|------|--------|---------|------|
| `chunk_size` | 500 | **200 - 300** | 小切片语义更聚焦，减少多主题混合 |

## 三、参数调整对照表（快速参考）

| 症状 | 调整方向 | 具体参数 | 风险 |
|------|---------|---------|------|
| 命中率 < 60% | 降低阈值 | `score_threshold` 0.5 → 0.3 | 误命中增加 |
| 命中率 < 60% | 减小切片 | `chunk_size` 500 → 300 | 语义完整性下降 |
| 命中率 < 60% | 切换检索 | 混合 → 全文 | 语义查询变差 |
| 误命中率高 | 开启 Rerank | `reranking_enable` = True | 响应时间增加 |
| 误命中率高 | 提高阈值 | `score_threshold` 0.5 → 0.7 | 召回率下降 |
| 误命中率高 | 减小切片 | `chunk_size` 500 → 200 | 上下文信息减少 |
| 中英文混排召回差 | 补充术语表 | 新增中英对照文档 | 知识库膨胀 |

## 五、调优工作流

```
1. 运行 verify_retrieval.py → 获取基线命中率
        │
2. 分析未命中查询 → 判断是召回问题还是精度问题
        │
3. 按优先级调整 1 个参数 → 重新运行验证
        │
4. 对比调整前后的命中率变化
        │
   ┌────┴────┐
   ▼         ▼
 改善      无改善/变差
   │         │
   ▼         ▼
 保持调整   回滚，试下一个参数
   │
   ▼
5. 达到目标命中率（≥ 70%）→ 完成
```

### 目标指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 整体命中率 | ≥ 70% | 10 个查询中至少 7 个 score ≥ 0.5 |
| 场景 1 命中率 | ≥ 90% | 知识库应覆盖的查询，必须高命中 |
| 场景 2 命中率 | 0% 可接受 | 联网兜底场景，知识库未命中是预期 |
| 场景 3 命中率 | ≥ 80% | 故障案例集应覆盖 |
| 平均最高 Score | ≥ 0.6 | 整体检索质量 |

---

## 六、⚠️ 注意事项

1. **每次只调整一个参数**：同时改多个参数无法判断哪个有效
2. **调整后重新运行验证**：用 `verify_retrieval.py` 对比调整前后
3. **重新分段耗时较长**：调整 `chunk_size` 需要重新向量化整个知识库
4. **保留基线报告**：每次调整前保存当前报告，便于回滚
5. **不改 PRD 锁定项**：`bge-large-zh-v1.5` 和 Weaviate 不可更换，只能在切片和检索参数上调优
